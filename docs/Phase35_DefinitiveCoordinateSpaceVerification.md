       = (0pt - 0pt) × 4.1667

**Verdict:** The actual content in the PNG starts at the same position where the legacy DB ratios predict field A1 should be. **Phase 31A's coordinate formula is missing the printed origin offset by approximately 852px horizontally and 1264px vertically.**

## Part 5 — PDFtoImage Crop Behavior

PDFtoImage (which wraps PDFium) renders the PDF page using the **default page rectangle**, which varies by PDF:

- If a **CropBox** is defined, PDFium renders the CropBox
- If **no CropBox** is defined, PDFium renders the **MediaBox**
- ExportAsFixedFormat produces a PDF with **only MediaBox** set (no CropBox, TrimBox, or BleedBox)

Therefore, **PDFtoImage renders the full MediaBox** — which is 612×792 pt for Letter. The resulting bitmap is full page at the specified DPI.

**This is definitively confirmed** by the measured PNG dimensions of 2550×3299 px = 612×(300/72) × 792×(300/72).

## Part 7 — Legacy Runtime Comparison

From the decompiled source and `TabletPipeline.md`:

**Legacy runtime (Tablet/i-Reporter):**
1. Renders PDF to bitmap at DpiPdfToImage (often 96 DPI for display)
2. Displays bitmap on WPF `Image` with `Stretch=None`
3. Canvas sized to match bitmap dimensions
4. For each cluster: `left_px = left_ratio × canvasWidth`
5. `canvasWidth = pageWidth_pt × DPI/72` = same as PNG pixel dimensions

**The legacy formula expands to:**
```
left_px = ((cell_left_pt + printedOriginXPt) / pageWidthPt) × (pageWidthPt × DPI/72)
        = (cell_left_pt + printedOriginXPt) × DPI/72
```

This is **identical** to our formula when printedOrigin is included:
```
leftPx = (printedOriginXPt + cellLeftPt - printAreaOriginLeftPt) × DPI/72
```

## Part 9 — Final Verdict

### Is the PNG a full page or print area only?

**Definitive answer: FULL PAGE.** 
- PNG dimensions: 2550×3299 px = 612×(300/72) × 792×(300/72) = Letter page size at 300 DPI
- Content position: measured at (852, 1264)px, NOT at (0, 0)
- PDF MediaBox: 612×792 pt (full Letter page)
- ExportAsFixedFormat with IgnorePrintAreas=false renders the print area **positioned within the full page** — it does not crop to the print area

### Should runtime coordinates be cellOffset or printedOrigin + cellOffset?

**Definitive answer: printedOrigin + cellOffset.**
- The printed origin (margins + centering) is **already baked into the PNG by ExportAsFixedFormat**
- Our fields must align with the printed content in the PNG
- CellOffset alone gives position relative to print area origin, which is at ~852px into the PNG
- Formula: `leftPx = (printedOriginXPt + cellLeftPt - printAreaOriginLeftPt) × DPI/72`

### Does Phase 34 correctly identify the missing printed origin?

**Definitive answer: YES.**
- Phase 34 computed: printedOriginXPt ≈ 206pt from page edge
- Empirically measured: content starts at 204.5pt (852px / 4.1667) from page edge
- Phase 34 prediction: 858px at 300 DPI
- Empirically measured: 852px at 300 DPI
- The ~6px (1.5pt) difference is within measurement tolerance (pixel scanning detects gridlines vs bordered content)

### Will adding printedOrigin fix alignment?

**Definitive answer: YES.**
- Adding ~852px to current field X coordinates and ~1264px to Y coordinates will place fields exactly over the cells in the PNG
- The formula: Cell A1 at (0+852, 0+1264) = (852, 1264)px — matches measured content position
- The formula: Cell C1 at (400+852, 0+1264) = (1252, 1264)px — next column group position
- These match the legacy DB ratio predictions
