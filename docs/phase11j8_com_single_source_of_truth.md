# Phase 11J.8 — Eliminate Coordinate Duplication (COM as the Single Source of Truth)

**Date:** July 10, 2026
**Status:** ✅ Complete

---

## Summary

The PNG background generated by Microsoft Excel COM was already pixel-perfect. The remaining problem was that the Runtime HTML overlay was still using a second coordinate engine (OpenXML + GeometryBuilder + CoordinateEngine), which estimated positions instead of using Excel's actual measurements.

This phase eliminates that duplication. **Excel COM is now the single source of truth** for both PNG rendering and HTML overlay coordinates.

---

## Architecture Change

### Before (Two Parallel Pipelines)

```
Excel Workbook
      │
      ▼
Excel COM ──→ PNG (correct)
      │
      ▼
OpenXML ──→ GeometryBuilder ──→ CoordinateEngine ──→ Runtime JSON ──→ HTML Overlay (incorrect)
```

### After (Single Pipeline)

```
Excel Workbook
      │
      ▼
Excel COM ──→ PNG
      │
      └──→ CaptureResult.Fields (exact Range.Left/Top/Width/Height in pixels)
                 │
                 ▼
            RuntimeMetadataService.Save()
                 │
                 ▼
            Forms/{templateId}.runtime.json
                 │
                 ▼
            GET /api/form/runtime/{id}
                 │
                 ▼
            RuntimeMetadataService.Load()
                 │
                 ▼
            RuntimeForm (COM pixel coordinates)
                 │
                 ▼
            Frontend renders leftPx/topPx/widthPx/heightPx directly
```

---

## Files Changed

### New File: `ExcelAPI/ExcelAPI/Runtime/RuntimeMetadataService.cs`

A singleton service that persists and loads COM field rectangles as `Forms/{templateId}.runtime.json`.

**`Save(CaptureResult, templateId, formsDir, workbookName)`**
- Called once during `POST /api/form/from-excel`
- Stores final pixel coordinates from `ExcelField.Left/Top/Width/Height`
- These coordinates already include margin offsets, centering offsets, and scaling applied by the COM pipeline
- Schema includes: version, capturedAt, workbookName, dpi, scaleX/Y, pageWidth/HeightPx, sheets with fields

**`Load(templateId, formsDir)`**
- Called on every `GET /api/form/runtime/{id}`
- Reads `.runtime.json` and deserializes directly into `RuntimeForm`/`RuntimeField`
- Returns `null` on any failure → triggers OpenXML fallback

### Modified: `ExcelAPI/ExcelAPI/Program.cs`

```csharp
builder.Services.AddSingleton<RuntimeMetadataService>();
```

### Modified: `ExcelAPI/ExcelAPI/Controllers/FormController.cs`

1. **Constructor**: Injects `RuntimeMetadataService`
2. **`FromExcel()`**: Calls `_runtimeMetadata.Save(captureResult, templateId, formsDir, file.FileName)` after COM capture
3. **`GetRuntime()`**: **Tries COM metadata FIRST**:
   - Calls `_runtimeMetadata.Load(templateId, formsDir)`
   - If a `RuntimeForm` is returned, returns it immediately
   - No `OpenXmlParser`, no `GeometryBuilder`, no `CoordinateEngine`, no `FieldDetector`
   - Falls back to OpenXML pipeline only if `.runtime.json` doesn't exist (legacy templates)

## Coordinate Pipeline Comparison

| Stage | COM Pipeline (✅ Active) | OpenXML Pipeline (⚠️ Fallback Only) |
|-------|--------------------------|-------------------------------------|
| Cell Left | `Range.Left` (exact) | `CharWidthToPoints()` (estimated) |
| Cell Top | `Range.Top` (exact) | Cumulative row heights (estimated) |
| Cell Width | `Range.Width` (exact) | `CharWidthToPoints()` (estimated) |
| Cell Height | `Range.Height` (exact) | Row height from XML (estimated) |
| Margins | Actual `PageSetup.LeftMargin` | Hardcoded `/ theme-derived` |
| Centering | `PageSetup.CenterHorizontally` applied | Calculated separately |
| Scale | `pngWidth / pageWidthPt` (actual render) | `DPI / 72` (theoretical) |
| Final Pixels | Stored as-is in `.runtime.json` | Recalculated on every GET |

## Remaining Minor Items

From code review:

1. **Hardcoded field styling** — `Save()` sets `fontSize = 11`, `bold = false`, `readOnly = false` for all fields. If `ExcelField` exposes actual values from the COM capture, these should be persisted instead.
2. **Sheet name hardcoded as `"Sheet1"`** — Should capture the actual worksheet name from the COM pipeline.
3. **WorkbookName with extension** — `file.FileName` includes `.xlsx`; consider using `Path.GetFileNameWithoutExtension()`.

These are non-blocking — the core COM-first architecture is correct.
