# Phase 5.3 — Forensic Investigation: Workbook Corruption Report

**Date:** 2026-07-20  
**Status:** Investigation Complete — Root Cause Identified  
**Action:** Do NOT fix yet — report first for approval

---

## Executive Summary

The generated workbook triggers Excel repair ("We found a problem with some content...") because `WorkbookValueWriter.PostProcessZipForConMas()` writes comments (`xl/comments1.xml`) but does NOT satisfy the OOXML contract required for comments to exist. The worksheet has no reference to the comments part, no VML drawing shapes exist for the comment popups, and no relationship chain connects the worksheet to the comments.

**Root cause:** Comments are treated as standalone XML rather than as part of a three-file contract (comments XML + VML drawing + worksheet `legacyDrawing` reference).

---

## Finding #1: Comments Without VML `legacyDrawing` (PRIMARY — Causes Excel Repair)

### Current Code Path

In `WorkbookValueWriter.cs`, `PostProcessZipForConMas()` calls `WriteConMasCommentsEntry()`:

```csharp
// Creates xl/comments1.xml in the ZIP
WriteConMasCommentsEntry(pkg, wbDef);
```

This writes a raw XML string to `xl/comments1.xml` with 6 ConMas-format cell comments. However, it does NOT:

1. **Add `<legacyDrawing r:id="rIdX">`** to `xl/worksheets/sheet1.xml`
2. **Create VML drawing files** (`xl/drawings/vmlDrawing1.vml`) for the comment popup shapes
3. **Add a VML relationship** in `xl/worksheets/_rels/sheet1.xml.rels`

### OOXML Comment Contract

For comments to work correctly in OOXML, three things must exist and be connected:

```
xl/comments1.xml                            ← Comment text content
     ↑ referenced by (worksheet → rels → comments part)
xl/worksheets/_rels/sheet1.xml.rels          ← Sheet-level relationships
     ↑ referenced by <legacyDrawing> in worksheet XML
xl/worksheets/sheet1.xml                     ← Worksheet with <legacyDrawing r:id="rIdX">
     ↑
xl/drawings/vmlDrawing1.vml                  ← VML shapes for comment popups
```

The current code only creates `xl/comments1.xml`. The complete chain is broken.

### Evidence

Validation of both `_edited (4).xlsx` and `_output (11).xlsx` shows:

```
[CORRUPTION] Comments part xl/comments1.xml exists but
             sheet1.xml has NO legacyDrawing reference!
```

This is the **exact** cause of "We found a problem with some content..." — Excel detects a comments part with no corresponding worksheet reference and enters repair mode to either create the reference or strip the comments.

### Original Workbook Has VML

The original workbook `FormTest - Copy.xlsx` has:
- `xl/comments1.xml` (6 legacy ConMas comments)
- `xl/drawings/vmlDrawing1.vml` (VML shapes for comment popups)

The legacy ConMas output properly creates BOTH parts. Our code restores the VML from the original (via ZIP restore which skips `xl/drawings/vmldrawing*`), but then rewrites `comments1.xml` without updating the worksheet's `legacyDrawing` reference or verifying the relationship chain.

## Finding #2: Missing Content Type Overrides (SECONDARY)

### Current Behavior

The ZIP restore skips `[Content_Types].xml`:

```csharp
bool isContentTypes = entryName.Equals("[content_types].xml", 
    StringComparison.OrdinalIgnoreCase);
if (isWorksheet || isSharedStrings || isComments || isVmlDrawing || isWorksheetRels || isContentTypes)
{
    skippedModified++;
    continue;  // ← KEEPS SDK-MODIFIED VERSION
}
```

Then `PostProcessZipForConMas.UpdateContentTypesForSheet3()` adds ONLY the sheet3 Override.

### Result

The validation script reports 11 `[MISSING]` Override entries. The SDK's modified `[Content_Types].xml` is missing entries that the original had, including:
- `/xl/workbook.xml`
- `/xl/worksheets/sheet1.xml`
- `/xl/worksheets/sheet2.xml`
- `/xl/styles.xml`
- `/xl/sharedStrings.xml`
- `/xl/theme/theme1.xml`
- `/xl/comments1.xml`
- `/docProps/core.xml`
- `/docProps/app.xml`

Additionally, Default entries for `.xml` and `.rels` extensions are missing.

**Note:** This may not directly cause Excel repair (Excel is lenient about content types), but it violates the OOXML specification and could cause validation failures with Open XML SDK validator.

## Finding #3: Unused Printer Settings (MINOR)

The generated workbook has extra `xl/printerSettings/printerSettings2.bin` and `printerSettings3.bin` entries created by the SDK. These are harmless but unnecessary — they remain because the ZIP restore does not explicitly remove them.

## Root Cause Timeline

| Phase | What Changed | Impact |
|-------|-------------|--------|
| Phase 5.4 | `WriteConMasCellComments()` added — writes comments via SDK (correctly handles VML rels) | Comments work |
| Phase 5.5.2 | `WriteConMasCellComments` DISABLED, replaced by `WriteConMasCommentsEntry()` in `PostProcessZipForConMas` | Comments written but VML/rels NOT updated |
| Phase 5.3 | `PostProcessZipForConMas` extended to create ExcelOutputSetting sheet | Sheet3 added, but no VML fix |

The regression was introduced in **Phase 5.5.2** when the SDK-based comment writer (`WriteConMasCellComments`) was disabled and replaced with the raw-ZIP writer (`WriteConMasCommentsEntry`).

The SDK-based writer (`WriteConMasCellComments`) properly handles the VML relationship chain because `WorksheetCommentsPart` and `LegacyDrawingPart` are SDK-managed types. The raw-ZIP approach bypasses the SDK entirely and must manually maintain the relationship chain.

---

## Concrete Fix Plan

### Fix #1: Update Sheet XML and Rels for Comments (REQUIRED)

In `WriteConMasCommentsEntry()` or a new `UpdateSheetForComments()` method, after writing `comments1.xml`:

1. **Parse sheet1.xml** from the ZIP
2. **Add `<legacyDrawing r:id="rIdX">`** child to the worksheet element
3. **Parse or create `xl/worksheets/_rels/sheet1.xml.rels`**
4. **Add relationship** for VML drawing: `rId="rIdX" Type="http://schemas.openxmlformats.org/officeDocument/2006/relationships/vmlDrawing" Target="../drawings/vmlDrawing1.vml"`
5. **Verify VML exists** — if original has `vmlDrawing1.vml`, it was preserved by the ZIP restore skip. If not, create a minimal VML drawing file.

### Fix #2: Restore [Content_Types].xml (RECOMMENDED)

In the ZIP restore, do NOT skip `[Content_Types].xml`. Instead:
1. Restore original `[Content_Types].xml`
2. Then in `PostProcessZipForConMas`, add the sheet3 Override to the restored version

This ensures all original Override and Default entries are preserved.

### Fix #3: Clean Up Unused Printer Settings (OPTIONAL)

Remove `printerSettings*.bin` entries that were added by the SDK but don't correspond to actual printer settings in the original workbook.
