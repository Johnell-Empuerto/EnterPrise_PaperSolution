## 3. Full Rendering Specification

### 3.1 Rendering Pipeline (Production)

```
Stage 0:  Workbook Load
          │
          ├── Parse .xlsx via OpenXML (DocumentFormat.OpenXml)
          │     ├── Read workbook.xml (sheets, metadata)
          │     ├── Read styles.xml (fonts, fills, borders)
          │     ├── Read sharedStrings.xml (text content)
          │     └── Read each worksheet.xml
          │
Stage 1:  Worksheet Selection
          │     ├── Select first visible sheet (or specified sheet)
          │     └── Determine print area (or fallback to UsedRange)
          │
Stage 2:  Page Setup Resolution
          │     ├── Paper size → page dimensions (pt)
          │     ├── Margins → printable area
          │     ├── Orientation → page orientation
          │     ├── Zoom/FitToPages → scaling
          │     └── CenterHorizontally/Vertically → centering offset
          │
Stage 3:  Column Width Calculation
          │     ├── Read <cols> from worksheet.xml
          │     ├── Convert character units → points (ECMA-376)
          │     ├── Read defaultColWidth from sheetFormatPr
          │     ├── Read Normal font from styles.xml → maxDigitWidth
          │     └── Compute cumulative column positions
          │
Stage 4:  Row Height Calculation
          │     ├── Read row heights from worksheet.xml
          │     └── Apply default row height (15pt or custom)
          │
Stage 5:  Merged Cell Resolution
          │     ├── Read <mergeCells> from worksheet.xml
          │     ├── Compute merged cell bounding rectangles
          │     └── Identify anchor cell vs merged region
          │
Stage 6:  Fill Rendering
          │     ├── Read fills from styles.xml
          │     ├── Apply background color to each cell/merge
          │     ├── Handle pattern fills (solid, gradient, pattern)
          │     └── Layer: all fills before borders and text
          │
Stage 7:  Border Rendering
          │     ├── Read borders from styles.xml
          │     ├── For each cell/merge range:
          │     │     ├── Top border (style, color, weight)
          │     │     ├── Bottom border
          │     │     ├── Left border
          │     │     └── Right border
          │     ├── Border priority: diagonal > outline > inner
          │     ├── Border overlap resolution (merge adjacent same-style)
          │     └── Border collapse (Excel behavior: adjacent cells share borders)
          │
Stage 8:  Font & Text Layout
          │     ├── Read fonts from styles.xml
          │     ├── For each cell:
          │     │     ├── Resolve font name, size, bold, italic, color
          │     │     ├── Apply horizontal alignment (left, center, right, fill)
          │     │     ├── Apply vertical alignment (top, center, bottom)
          │     │     ├── Apply wrap text
          │     │     ├── Apply text rotation
          │     │     ├── Apply indent
          │     │     └── Measure and render text within cell bounds
          │     └── Handle text overflow (clip vs wrap vs shrink-to-fit)
          │
Stage 9:  Image & Shape Rendering
          │     ├── Read images from worksheet drawings
          │     ├── For each image:
          │     │     ├── Decode image data
          │     │     ├── Position within cell/page bounds
          │     │     └── Apply scaling / aspect ratio
          │     ├── Read shapes from worksheet drawings
          │     └── Render shape objects (text boxes, controls, etc.)
          │
Stage 10: Gridline Rendering
          │     ├── Read ShowGridlines / ShowHeadings from sheet
          │     └── Draw gridlines (default: light gray, 0.5pt)
          │
Stage 11: Coordinate Mapping
          │     ├── Compute printable origin (margin + centering)
          │     ├── Map cell coordinates to page coordinates
          │     └── Apply scale factor (DPI / 72)
          │
Stage 12: Canvas Rendering
          │     Draw all elements to SkiaSharp canvas:
          │     ├── Fills (Layer 1)
          │     ├── Gridlines (Layer 2)
          │     ├── Borders (Layer 3)
          │     ├── Text (Layer 4)
          │     ├── Images/Shapes (Layer 5)
          │     └── Cluster overlays (Layer 6 — debug only)
          │
Stage 13: PDF Generation
          │     ├── Create SkiaSharp PDF document
          │     ├── Render each page at printer DPI
          │     └── Embed fonts, metadata
          │
Stage 14: PNG Generation
          │     ├── Render at 300 DPI for production
          │     └── Render at 72 DPI for preview
          │
Stage 15: Field Extraction
          │     ├── Read cell comments
          │     └── Extract field metadata
          │
Stage 16: Database Storage (Legacy compatibility)
          │     ├── Store as normalized ratios
          │     └── Include metadata for re-rendering
          │
Stage 17: Runtime Rendering
          │     From stored data:
          │     ├── Load background PNG
          │     ├── Render cluster overlays at stored positions
          │     └── Provide zoom, pan, selection UI
```

### 3.2 Rendering Order (Z-Index)

| Layer | Content | Z-Index |
|:-----:|:--------|:-------:|
| 1 | White page background | 0 |
| 2 | Cell fills (pattern fills, solid colors) | 1 |
| 3 | Drawing fills (shape backgrounds, image bitmaps) | 2 |
| 4 | Gridlines (if visible) | 3 |
| 5 | Cell borders (top, bottom, left, right, diagonal) | 4 |
| 6 | Drawing borders (shape outlines) | 5 |
| 7 | Cell text content | 6 |
| 8 | Drawing text content (text boxes, labels) | 7 |
| 9 | Images (bitmap renders) | 8 |
| 10 | Cluster overlays (debug/annotation only) | 9 |

### 3.3 Algorithm for Each Stage

#### Stage 3 — Column Width to Points (ECMA-376)

```
Input:  charWidth (from <col width="...">)
        maxDigitWidth (from Normal font, default 7.33)

Process:
  pixelWidth = charWidth * maxDigitWidth + 5.0  // 5px padding constant
  pointWidth = pixelWidth * 72.0 / 96.0          // screen DPI → points

Edge Case: defaultColWidth in sheetFormatPr overrides 8.43 char default
Edge Case: Hidden columns (hidden="1") excluded from width
Edge Case: customWidth="1" indicates explicitly set width, NOT auto-fit
```

#### Stage 7 — Border Rendering

```
For each cell (or merged cell range):
  1. Read border entries from styles.xml cellXf
  2. For each side (top, bottom, left, right):
     a. Get border style: thin, medium, thick, dotted, dashed, double, etc.
     b. Get border color: indexed, theme, or RGB
  3. Draw borders in this priority:
     a. Diagonal up/down (overlapping)
     b. Right + bottom (Excel draws shared borders from the right/bottom cell)
     c. Left + top (less priority — overwritten by adjacent cell)
  4. Border weight mapping:
     thin    → 0.5pt
     medium  → 1.0pt
     thick   → 2.0pt
     double  → 2.0pt (two 0.5pt lines with 0.5pt gap)
     hair    → 0.25pt
     dotted  → 0.5pt (dotted)
     dashed  → 0.5pt (dashed)

Key Difference from CSS: Excel draws RIGHT border for column A,
LEFT border for column B at the same position. The visual result
is the one with higher priority (usually the right/bottom border
of the preceding cell, unless explicitly overridden).
```

#### Stage 8 — Text Layout

```
For each cell:
  1. Resolve font (normal style + fontId from cellXf)
  2. If merged, use merged cell bounds instead of individual cell
  3. Apply text rotation:
     - 0°: normal horizontal
     - 90°: vertical (top-to-bottom)
     - -90°: vertical (bottom-to-top)
     - 0° < |angle| < 90°: custom rotation
  4. Apply horizontal alignment:
     - general: strings left, numbers right
     - left: left-align within cell padding (2pt)
     - center: center within cell
     - right: right-align within cell padding
     - fill: repeat text to fill width
     - justify: space text across width
     - centerContinuous: center across selection (for merged cells)
  5. Apply vertical alignment:
     - top: text at top with 1pt padding
     - center: vertically centered
     - bottom: text at bottom with 1pt padding (default)
  6. Wrap text:
     - If wrapText = true: break lines at word boundaries
     - If wrapText = false: clip or overflow (depending on adjacent cells)
  7. Text clipping:
     - If text exceeds cell width: clip at cell boundary (default)
     - Shrink-to-fit if enabled: reduce font size until text fits
     - Overflow to next cell if adjacent cell is empty and not merged
```

## 5. Implementation Roadmap — 10 Milestones

### Milestone 1 — Visual Rendering Parity (Background Fills)

**Goal:** Render cell background fills identically to Excel.

**Tasks:**
1. Parse `<fills>` from `styles.xml` (DocumentFormat.OpenXml)
2. Map fill types: solid, pattern (gray125, gray0625), gradient, none
3. Resolve indexed colors (56-color palette) + theme colors + RGB
4. Render fills on SkiaSharp canvas for each cell/merge range
5. Compare against Excel PDF (side-by-side with `compare-fills` tool)

**Files to modify:**
- `ExcelAPI/ExcelAPI/Generators/PreviewGenerator.cs` (add fill rendering)
- `ExcelAPI/ExcelAPI/Services/ExcelCaptureService.cs` (add OpenXML fill parser)
- `paperless/lib/formDefinition.ts` (add fill style types)

**Risks:**
- Theme color resolution requires theme XML parsing
- Pattern fill rendering requires SkiaSharp pattern brushes

**Validation:**
- Extract fill colors from XLSX, compare against Excel PDF pixel scan
- 100% match on solid fills for all 10 representative forms

**Exit criteria:** All visible fills in representative forms match Excel to ±2 ΔE.

---

### Milestone 2 — Border Engine

**Goal:** Render all cell borders identically to Excel.

**Tasks:**
1. Parse `<borders>` from `styles.xml`
2. Implement all 14 border styles (thin, medium, thick, double, dotted, dashed, hair, etc.)
3. Implement border color resolution (indexed, theme, RGB, auto)
4. Implement Excel's border priority rules (right/bottom cell wins)
5. Implement diagonal borders (up, down, both)
6. Implement border collapse (adjacent cells share border lines, not double-draw)
7. Compare against Excel PDF border render

**Files to modify:**
- New: `ExcelAPI/ExcelAPI/Rendering/BorderEngine.cs`
- `ExcelAPI/ExcelAPI/Generators/PreviewGenerator.cs` (integrate border rendering)
- `paperless/lib/formDefinition.ts` (add border types)

**Risks:**
- Double borders are two lines with a gap — requires special handling
- Diagonal borders with merged cells

**Validation:**
- Side-by-side: renderer vs Excel PDF at 300 DPI
- Border thickness ±0.5px, color ±5 ΔE

**Exit criteria:** All borders in representative forms match Excel visually.

---

### Milestone 3 — Fill Engine (Complete)

**Goal:** Complete fill rendering with gradients, patterns, and all edge cases.

**Tasks:**
1. Implement gradient fills (linear, path, rectangular)
2. Implement pattern fills (18 Excel patterns)
3. Implement conditional formatting fills

**Exit criteria:** Fills match Excel PDF exactly for all fill types across 457 forms.

---

### Milestone 4 — Text Layout Engine

**Goal:** Render all cell text identically to Excel.

**Tasks:**
1. Parse `<fonts>` from `styles.xml` (font name, size, bold, italic, color, underline, strikethrough)
2. Parse `<si>` from `sharedStrings.xml` (rich text formatting)
3. Implement horizontal alignment (left, center, right, fill, justify, centerContinuous, distributed)
4. Implement vertical alignment (top, center, bottom, justify, distributed)
5. Implement wrap text (word-wrap within cell bounds)
6. Implement shrink-to-fit (reduce font size until text fits)
7. Implement text rotation (-90°, 0°, 90°, custom angles)
8. Implement indent (left padding in character increments)
9. Implement rich text (different formatting within same cell)
10. Compare against Excel PDF text render

**Files to modify:**
- New: `ExcelAPI/ExcelAPI/Rendering/TextEngine.cs`
- New: `ExcelAPI/ExcelAPI/Rendering/FontResolver.cs`
- `ExcelAPI/ExcelAPI/Generators/PreviewGenerator.cs`

**Risks:**
- Font availability differs between machines (use font fallback)
- Rich text layout requires multi-segment text measurement
- Wrap text with merged cells adds complexity

**Validation:**
- Extract text positions from Excel PDF, compare against renderer
- Text position ±1pt, font size ±0.5pt

**Exit criteria:** All visible text in representative forms matches Excel position and appearance.

---

### Milestone 5 — Image & Shape Engine

**Goal:** Render images and drawing objects identically to Excel.

**Tasks:**
1. Parse drawing XML (`xl/drawings/drawing1.xml`)
2. Extract image data from `xl/media/` (PNG, JPEG, etc.)
3. Implement image positioning (two-cell anchor, one-cell anchor, absolute)
4. Implement image scaling (aspect ratio, stretch)
5. Implement shape rendering (rectangles, text boxes, OLE objects)
6. Handle shape text content (text boxes with formatting)
7. Handle shape visibility (visible vs hidden)

**Files to modify:**
- New: `ExcelAPI/ExcelAPI/Rendering/ImageEngine.cs`
- New: `ExcelAPI/ExcelAPI/Rendering/ShapeEngine.cs`
- `ExcelAPI/ExcelAPI/Generators/PreviewGenerator.cs`

**Risks:**
- Embedded images may use different formats (EMF, WMF)
- Shape coordinates use EMU (English Metric Units) — 1 EMU = 1/36000 cm

**Exit criteria:** Images and shapes match Excel PDF within 2px at 300 DPI.

---

### Milestone 6 — Coordinate Engine

**Goal:** Compute exact cell-to-page coordinate mapping.

**Tasks:**
1. Compute cumulative column widths from `<cols>` data
2. Compute cumulative row heights from `<row>` data
3. Apply print area origin (first column/row in print area)
4. Apply page margins and centering
5. Apply scaling (zoom, FitToPages)
6. Handle merged cell rectangles (use MergeArea bounds)
7. Map cell coordinates to page coordinates
8. Compare against coordinates from Phase 4 PDF ground truth

**Files to modify:**
- New: `ExcelAPI/ExcelAPI/Rendering/CoordinateEngine.cs`

## 6. Development Rules

### 6.1 Architecture Rules

1. **Continue from existing codebase** — do not redesign from scratch.
2. **Prefer improving existing modules** over replacing them.
3. **Keep the renderer modular** (one engine per concern: fills, borders, text, images, coordinates).
4. **Maintain compatibility** with current Next.js frontend and ASP.NET Web API backend.
5. **Abstract new rendering behind `IWorksheetDataProvider`** for parallel operation alongside COM.

### 6.2 Rendering Rules

1. **Excel is the visual source of truth** — when rendering differs, fix the renderer.
2. **Use the legacy database ONLY as a validation reference**, not as a design constraint.
3. **Pixel-level parity** means: same position, same size, same color, same border weight as Excel.
4. **Do NOT hardcode constants** derived from legacy ConMas behavior (e.g., 51.02pt margin, 7.33 maxDigitWidth, 50.1pt/col default).
5. **Read all data from the XLSX file** (not from COM, not from XML copy).

### 6.3 Development Process Rules

1. **Each milestone must be independently testable** against Excel PDF output.
2. **Add validation alongside implementation** — don't wait for Milestone 9.
3. **Use the Phase 4 PDF ground truth methodology** for all coordinate validation.
4. **Run the OpenXML renderer in parallel** with the COM pipeline until Milestone 9.
5. **Log all rendering decisions** that differ from Excel for audit trail.
6. **No more formula brute-forcing** (Phase 9 approach). The algorithm is understood — implement it correctly.

### 6.4 COM Minimization Rules

1. **Use OpenXML wherever a data source exists** — it's faster, safer, and thread-safe.
2. **Keep COM for:**
   - Generating output XLSX workbooks (end-user files)
   - Generating reference PDFs for visual comparison
   - Edge-case workbooks that OpenXML cannot parse
3. **Abstract COM behind an interface** so it can be disabled or replaced.
4. **Never read rendering data from COM** — only from OpenXML or cached values.
