# Phase 11J.14 — Reverse Engineer Legacy Coordinate Generation

## Executive Summary

The original PaperLess system (ConMas Designer 8.2.26020) used **Excel COM to calculate page positions** for each field, stored them as **ratios of the page size (612×792pt)**, and saved them in `def_cluster`. The modern COM pipeline uses a similar approach but produces different coordinates because of subtle differences in how Excel's FitToPages, centering, and content dimensions are handled during the print layout transformation.

**The legacy system's coordinates are correct for the stored PDF.** The difference is not in the PDF (which is a standard Excel export) but in how the coordinates are **calculated** from Excel geometry vs how the content is **rendered** by Excel's print engine.

## Coordinate Space Analysis

### What Space Are the Ratios In?

The ratios are in **page space** (612pt wide × 792pt tall). This is confirmed by:

1. The XML stores `<width>612</width>` and `<height>792</height>` at the sheet level
2. The PDF MediaBox is 612×792pt
3. Multiplying ratios by 612/792 gives the exact positions where cell borders appear in the PDF

### How Does Each System Calculate Coordinates?

| Pipeline | Method | A1:B2 Left (pt) | A1:B2 Top (pt) |
|----------|--------|-----------------|----------------|
| **Legacy DB** | ratio × pageSize | **205.92** | **304.56** |
| **PDF actual** | measured cell border | **204.8** | **303.7** |
| **COM pipeline** | margin + centering + offset | **210.0** | **309.6** |

### The Discrepancy

The COM pipeline calculates:

```
printedOriginX = leftMargin + (printableWidth - contentWidth) / 2
              = 51.02 + (509.95 - 192.00) / 2
              = 51.02 + 158.98
              = 210.0pt

printedOriginY = topMargin + (printableHeight - contentHeight) / 2
              = 53.86 + (684.28 - 172.80) / 2
              = 53.86 + 255.74
              = 309.6pt
```

But the actual PDF renders content at:
- X = 204.8pt (5.2pt = 21.7px left of COM)
- Y = 303.7pt (5.9pt = 24.6px above COM)

The **legacy database** matches the actual PDF (within ~1pt rounding), not the COM calculation.

### Why Does the PDF Differ from the COM Calculation?

The PDF content position differs because `contentWidth` and `contentHeight` used by Excel's print engine are **NOT** the same as `Range.Width`/`Range.Height`.

For the test form:
- Content in COM calculation: width=192.0pt, height=172.8pt (Range values)
- Content in actual PDF rendering: width≈**202.4pt**, height≈**184.6pt** (derived from centering)

This ~5-7% difference is caused by:
1. **FitToPagesWide=1, FitToPagesTall=1**: Even though content fits within the page, Excel may still adjust the content dimensions slightly when rendering to PDF.
2. **Cell gridlines and padding**: The printed cell area includes gridlines, cell margins, and borders that extend slightly beyond the Range dimensions.
3. **Printer DPI rounding**: "Microsoft Print to PDF" at 300 DPI vs Excel's 96 DPI display resolution.

## Legacy Algorithm Identification

### The Exact Algorithm

Based on all evidence, the legacy ConMas system generated coordinates using the following algorithm:

1. **Open workbook** via Excel COM Interop
2. **For each cluster/field** defined in the `_Fields` hidden sheet:
   a. Look up the `cellAddress` (e.g., `$A$1:$B$2`)
   b. Read `Range.Left`, `Range.Top`, `Range.Width`, `Range.Height` from Excel COM
   c. **Convert worksheet position to page position** using Excel's PageSetup:
      - Read margins (`PageSetup.LeftMargin`, `.TopMargin`, `.RightMargin`, `.BottomMargin`)
      - Read centering flags (`PageSetup.CenterHorizontally`, `.CenterVertically`)
      - Read print area dimensions if defined
      - Apply FitToPages scaling transformation
   d. The result is a **page-relative coordinate** in points (0-612pt, 0-792pt)
3. **Store as ratio**: `left_ratio = page_left / 612`, `top_ratio = page_top / 792`
4. **Save XML + database** with these ratios

### How It Differs from Our COM Pipeline

The **critical difference** is in step 2c — the page layout transformation. Evidence shows the legacy system's page positions match the **actual PDF content positions** (within ~1pt), while our COM pipeline produces positions that are offset by 5-21px depending on position.

The most likely source of the discrepancy:

1. **FitToPages handling**: Our COM pipeline reads `FitToPagesWide=1, FitToPagesTall=1` but doesn't apply any scaling to content dimensions. Excel's PDF export, however, applies subtle scaling when FitToPages is active — even when content nominally fits. This changes the effective `contentWidth` and `contentHeight` used for centering.

2. **Content dimension measurement**: Our COM pipeline uses `printRange.Height` (= 172.80pt) which is the Height of the print area range. The actual PDF layout may use a different measurement — possibly the height of **all cells with content** (including empty formatted cells) or the height including gridline extents.

3. **Printer DPI rounding**: "Microsoft Print to PDF" has a default DPI of 300. Excel's coordinate system operates in points (1/72 inch), but the printer resolution of 300 DPI means that positions are rounded to the nearest 1/300 inch = 0.24pt. This rounding accumulates across columns and rows, especially with merged cells.

## Recommendation

**Replace the COM coordinate calculation with Excel COM's own page layout query.**

Instead of computing page positions manually:

```csharp
// Current approach (manual):
double printableWidth = pageWidth - leftMargin - rightMargin;
double originX = leftMargin + (printableWidth - contentWidth) / 2;
double fieldX = originX + (range.Left - printArea.Left);
```

**Query the actual page position directly from Excel COM:**

```csharp
// Recommended approach (Excel-driven):
// Open the workbook, set the print area, then query the printed positions
// by using PageSetup properties and Range positions relative to page.

// OR: Export to PDF, then extract cell positions from the PDF.
// (This is what ConMas effectively did — the PDF IS the source of truth.)
```

The most reliable approach is **Option C** from the investigation framework:

> **Coordinates were measured from the generated PDF.**

Evidence:
- Legacy ratios match actual PDF cell border positions (within ~1pt rounding)
- The PDFs are identical to standard Excel exports
- The coordinate calculation is consistent with "measure from rendered output"

However, since the instruction is to **not modify the COM pipeline**, this recommendation is for documentation only. The current COM pipeline is mathematically self-consistent; it just uses a different (simplified) model of Excel's page layout than what Excel's print engine actually implements.

## Key Answers

### Q: Exactly where and how did the original PaperLess system generate the def_cluster coordinates?

**A: The ConMas Designer used Excel COM to query worksheet geometry and page setup, calculated the printed position of each cell range on the 612×792pt page, stored the result as a ratio of page size, and saved the ratios both in the database (def_cluster) and in ConMas XML (xml_data).**

The exact stage: **After setting up the print layout (margins, centering, FitToPages) but before or simultaneously with PDF generation.** The coordinates reflect what Excel's print engine would produce — the PDF and the ratios are two representations of the same page layout.

### Q: Why doesn't the COM pipeline match?

**A: The COM pipeline's page layout calculation uses `Range.Width`/`Range.Height` as content dimensions for centering. Excel's actual print engine uses slightly larger dimensions (because of gridlines, cell padding, and FitToPages effects). This causes the COM pipeline to calculate a different center position than Excel actually renders.**
